# crane_blog

A small static site generator whose Markdown-to-HTML transformation is
authored in [Rocq](https://rocq-prover.org) and extracted to C++ by
[Bloomberg's Crane](https://github.com/bloomberg/crane).

## What this is

The transformation core — frontmatter parsing, a subset of Markdown,
page assembly, and the rendered HTML shell — is written as total Rocq
functions in `src/Logic.v`. During `dune build`, the Crane plugin
extracts that development to a pair of C++ translation units (`blog.h`
and `blog.cpp`) and a minimal `main.cpp` is auto-generated by a rule in
`src/dune`. The three files are compiled with `clang++ -std=c++2b -O2`
into a single native binary, `blog_generator.exe`.

At runtime the binary is effectively a thin I/O driver. It reads
`./posts/` (relative to its working directory), parses any `*.md` files
found there, copies every non-`.md` file into `_site/posts/`, and
writes `_site/index.html`, `_site/styles/site.css`, and one
`_site/<slug>/index.html` per published post. All decisions about what
the HTML looks like — tag shape, escaping, inline parsing, heading
policy, post ordering, draft filtering — live in `Logic.v`.

We deliberately do not describe this as a "verified static site
generator". The verification surface is narrow and worth stating
honestly.

## What is and isn't checked

What holds by construction of the toolchain:

- **Totality.** Every Rocq `Fixpoint` in `Logic.v` recurses on
  decreasing structural arguments or on an explicit `nat` fuel
  parameter. The shared `fuel` notation is `4000` (`Logic.v:36`) and
  is reused by the character scanners, `render_inline_list`, and the
  slugifier; the block parser takes a separately-supplied bound. If
  Rocq accepts the file, every function terminates.
- **Type safety at the Rocq level.** `Logic.v` is type-checked by Rocq
  9.0 before anything else runs. `IO` is declared as a `Notation` for
  `itree (dirE +' ioE)` (`Logic.v:17`), which matters for extraction:
  it lets Crane's monad table see `itree` directly and avoid marking
  I/O-performing functions with `__attribute__((pure))`.
- **Extraction succeeds.** `Crane Extraction "blog" run.`
  (`Logic.v:721`) runs as part of `dune build`. If it fails (for
  example because something in the monad table got mis-registered) the
  build stops.
- **C++ compiles.** The generated C++ is compiled with clang at `-O2`,
  bracket depth raised to 1024. If extraction produces something the
  C++ frontend rejects, the build stops.

What is **not** proven and should not be assumed:

- There is no theorem stating that `parse_post` produces a semantically
  correct AST for any reasonable definition of "CommonMark" or even
  for the subset we do support.
- There is no theorem linking the `Block` / `Inline` AST to the HTML
  emitted by `render_block_impl`. In particular there is no proof of
  well-formedness of the output, nor of XSS-safety. We do call
  `html_escape` at every user-controlled site, but `html_escape`
  itself is unverified.
- All I/O is trusted. `read`, `write_file`, `list_directory`, and
  `create_directory` are Crane-provided effect handlers whose C++
  implementations are taken on faith.
- Recursion fuel is finite. A post long enough to exhaust 10 000 block
  iterations or a line long enough to exhaust 4 000 character
  iterations would be silently truncated. No input we ship comes
  anywhere near that.

In short: the core is total and the pipeline compiles; the *semantics*
of the pipeline are reviewed, not proven.

## Layout

```
.
├── dune-project              # project definition, pins rocq >= 9.0, rocq-crane
├── src/
│   ├── Logic.v               # the entire transformation core
│   └── dune                  # (coq.theory ...) + rule generating main.cpp
│                             # + rule linking blog_generator.exe
├── posts/                    # input: *.md and any assets they reference
├── _site/                    # output (gitignored; produced by the binary)
├── Dockerfile                # reproducible build: opam + Rocq 9.0 + Crane
└── .github/workflows/deploy.yml   # build image -> run -> upload to Pages
```

There is no hand-written C++ in the repository. `main.cpp` is emitted
by a `with-stdout-to` rule in `src/dune`; `blog.h` and `blog.cpp` are
emitted by the Crane plugin into the `FormalBlog` build directory.

## Build

### Docker (recommended)

The Dockerfile pins the exact combination that is known to work: opam
2.5, OCaml 5.4, Rocq 9.0.0, `coq-itree`, `coq-paco`, `coq-ext-lib`,
and `rocq-crane` installed from the upstream repository.

```bash
docker build -t crane-blog .
docker run --name crane-blog-run crane-blog
docker cp crane-blog-run:/home/opam/crane-blog/_site ./_site
docker rm crane-blog-run
```

The image's `CMD` runs `blog_generator.exe` once, which writes
`/home/opam/crane-blog/_site` inside the container and exits. We
deliberately do not pass `--rm` to `docker run`, because we need the
stopped container to still exist for `docker cp`. This is the same
sequence `.github/workflows/deploy.yml` uses.

### Inside the container (development)

For iterative work, build the image once and shell in:

```bash
docker build -t crane-blog .
docker run --rm -it -v "$PWD":/home/opam/crane-blog crane-blog bash
# inside:
eval $(opam env)
dune build src/blog_generator.exe
./_build/default/src/blog_generator.exe
```

The binary takes no arguments; `./posts` and `./_site` are hard-coded
relative paths in `run` (`Logic.v:707`).

## Writing posts

A post is a UTF-8 Markdown file in `posts/`. Optional YAML-ish
frontmatter is delimited by two `---` lines at the very top. The
parser in `parse_frontmatter_lines` (`Logic.v:497`) recognises
exactly these keys; everything else is silently ignored:

| key              | type    | used for                                             |
|------------------|---------|------------------------------------------------------|
| `title`          | string  | `<title>` and the page's sole `<h1>`                 |
| `date`           | string  | rendered verbatim; also the sort key (descending)    |
| `slug`           | string  | output directory name; falls back to the file stem   |
| `summary`        | string  | the post deck and index row subtitle                 |
| `topic`          | string  | the small meta chip in the rail and index row        |
| `lead_image`     | string  | filename in `posts/`; rendered above the post body   |
| `lead_image_alt` | string  | `alt` text for the lead image                        |
| `draft`          | `true`  | if exactly `true`, the post is dropped from the site |

If `title` is empty, the first `Heading2` or `Heading3` in the body is
used. If `slug` is empty or whitespace, the file stem (without `.md`)
is used; spaces, slashes, and dots in the slug become `-`.

### Supported Markdown

The parser is intentionally small. What it recognises:

- **Fenced code blocks.** Lines beginning with ``` ``` ``` open and
  close a `CodeBlock`. Anything after the opening fence is taken as a
  language tag and emitted as `data-lang="..."` on the inner
  `<code>`.
- **Headings.** Both `# ` and `## ` produce an `<h2>`; `### `
  produces an `<h3>`. We never emit a second `<h1>` from post
  content — the single `<h1>` on each post page is the frontmatter
  title, rendered by the page shell. See the heading-policy comment
  at `Logic.v:482`.
- **Images on their own line.** A line matching `![alt](src)` becomes
  a `<figure>` containing an `<img>`. `src` is resolved relative to
  `../posts/`, so `lead_image: hello_world.jpg` and `![…](hello_world.jpg)`
  both point at `posts/hello_world.jpg` after the build. Non-`.md`
  files in `posts/` are copied verbatim into `_site/posts/`.
- **Inline constructs.** `[label](url)` links, `` `code` `` spans,
  `*emphasis*`, and `**strong**`. Unterminated `*` or `**` emits a
  literal asterisk rather than swallowing input.
- **Paragraphs.** Any run of non-empty lines that does not match the
  above is joined with single spaces and parsed as inlines.

There is no list syntax, no block quote, no table, no HTML
pass-through, no reference-style links, no autolink, no
setext-style heading, and no hard line break. Adding any of these
means editing `Logic.v` and rebuilding.

## Deploy

`.github/workflows/deploy.yml` runs on every push to `main`. It
builds the Docker image on a stock `ubuntu-latest` runner, starts a
named container, `docker cp`s `_site` out, and publishes the tree
via `actions/upload-pages-artifact` + `actions/deploy-pages`. A cold
build takes roughly five minutes; subsequent pushes re-use the
registry layer cache.
